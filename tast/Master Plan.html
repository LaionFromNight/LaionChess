Implementation Master Plan

Opening Trainer

Roadmap for extending the existing LaionChess analysis app (Vite + React + TypeScript, GitHub Pages) with an opening training platform: course catalog, Learn/Practice trainer, course creator, and shared theming. The interactive prototype in this project is the design reference for every screen.

0 · Current state

  • Single-page analysis board: src/App.tsx (~1000 lines), Board.tsx, Square.tsx, MoveList.tsx, PromotionPicker.tsx, SpottingPanel.tsx.
  • Chess logic lives in src/chess/; PGN/FEN docs already written in tech-chess-docs/.
  • Styling: inline style objects + small App.css; colors hard-coded per component.
  • No routing, no persistence layer, Unicode glyph pieces only.

2 · Data model

type Ply = {
  san: string;            // "Nxd4"
  from: Square; to: Square;
  promo?: PieceType;
  note?: string;          // coach text (Learn mode, before user move)
};

type Line = {
  id: string;
  name: string;           // "Mieses Variation — 4...Nf6"
  tag?: 'Main' | 'Sideline' | 'Trap' | 'Punish';
  plies: Ply[];           // from the starting position
};

type Course = {
  id: string; name: string; description: string;
  author: string; playAs: 'w' | 'b';
  lines: Line[];
};

type Progress = {        // localStorage: `laionchess-progress:${courseId}`
  learn: Record<LineId, true>;
  practice: Record<LineId, true>;
  srs?: Record<LineId, { due: number; ease: number; reps: number }>;
};

Bundled courses ship as static JSON in public/courses/; user-created courses live in localStorage first, with export/import as .json or PGN. A backend (Supabase or similar) is intentionally out of scope until P6+ — everything above works fully client-side on GitHub Pages.

3 · Trainer state machine

idle ──start line──▶ prompt(user)
prompt ──correct move──▶ animate ──▶ opponent reply ──▶ prompt
prompt ──wrong move───▶ flash + mistake++ ──▶ prompt
prompt ──no plies left─▶ complete(line) ──▶ save progress ──▶ next line
  • Learn: coach note + arrow + source-square hint shown before every user move.
  • Practice: no hints; arrow only via the Hint button (marks the run imperfect); mistakes counted.
  • Validation is repertoire-based (expected move only) — no engine needed in the trainer; full legality stays in the analysis app.

4 · Theming & settings

  • One settings store drives the whole app: boardTheme (Classic Wood / Neon Night / Forest / Ice), pieceSet (Cburnett / Merida / Alpha / Glyph), accent (cyan / green / magenta / amber), arrows, coords.
  • Board colors are CSS custom properties on the board root — adding a theme is a 4-line entry, no component changes.
  • Piece sets resolve to public/pieces/<set>/<code>.svg; the Glyph set keeps the current Unicode look as a zero-asset fallback.
  • The settings popover (gear) is shared between the header and the trainer toolbar — same component, same store.

5 · Risks & decisions

  4. Lighthouse: no regressions vs. the current analysis app.
